## Описание проекта "Веб-страницы Каталога"

Данное описание отражает этапы создания и настройки Django-проекта 
"Веб-страницы Каталога", включающего приложение `catalog`.

**1. Git и организация разработки:**

*   Создан файл `.gitignore` для исключения ненужных файлов из отслеживания (например, файлов виртуального окружения, файлов `.pyc`, файлов `.sqlite3`, папки `__pycache__` и т.д.). Файл настроен с использованием онлайн-генератора или вручную.
*   Организована разработка в соответствии с моделью GitFlow:
    *   Созданы основные ветки `main` и `develop`.
    *   Для решения задач (в частности, для выполнения домашних заданий) создавались отдельные feature-ветки, ответвляющиеся от `develop` и имеющие префикс `feature/`.

**2. Оформление проекта и документация:**

*   Создан и оформлен файл `README.md` с подробным описанием проекта "Веб-страницы Каталога". Файл содержит информацию о структуре проекта, используемых соглашениях по именованию и ссылки на документацию.
*   Сгенерирован файл `requirements.txt` с перечнем зависимостей проекта.  Для создания файла использована команда `pip freeze > requirements.txt` внутри активированного виртуального окружения.

**3. Создание и настройка приложения `catalog`:**

*   Создано новое Django-приложение под названием `catalog` с помощью команды `python manage.py startapp catalog`.
*   Приложение `catalog` зарегистрировано в настройках проекта (в `INSTALLED_APPS` в файле `BASE_DIR/settings.py`). В список `INSTALLED_APPS` добавлена строка `'catalog.apps.CatalogConfig'`, указывающая на конфигурацию приложения.
*   Настроена маршрутизация для приложения:
    *   Создан файл `urls.py` в директории приложения `catalog`.
    *   В главном файле `BASE_DIR/urls.py` проекта добавлено подключение маршрутов приложения `catalog` с использованием функции `path` и `include`.

**4. Создание шаблонов:**

*   Создана директория `templates` внутри директории приложения `catalog`. В файле `settings.py` указан путь к этой директории для Django-шаблонизатора.
*   Внутри директории `templates` созданы два HTML-шаблона:
    *   `home.html` - шаблон для домашней страницы.
    *   `contacts.html` - шаблон для страницы с контактной информацией.
*   Шаблоны стилизованы с использованием CSS-фреймворка Bootstrap.  Подключение Bootstrap CSS (и JavaScript, если необходимо) выполнено через CDN в шаблонах.
*   Обеспечена базовая структура HTML-документов, включающая `<!DOCTYPE html>`, `<html lang="ru">`, `<head>` и `<body>`.

**5. Реализация контроллеров (Views):**

*   В файле `views.py` приложения `catalog` созданы два контроллера (view functions):
    *   Контроллер для отображения домашней страницы (`home_view` или аналогичное название). Этот контроллер возвращает отрендеренный шаблон `home.html` с помощью функции `render`.
    *   Контроллер для отображения страницы с контактной информацией (`contacts_view` или аналогичное название). Этот контроллер возвращает отрендеренный шаблон `contacts.html` с помощью функции `render`.

**6. Настройка маршрутизации для контроллеров:**

*   В файле `urls.py` приложения `catalog` настроена маршрутизация для контроллеров с использованием функции `path`:
    *   Определен URL-путь для домашней страницы (например, пустая строка `''`, `'home/'`, `'/'`), который связан с контроллером `home_view`.
    *   Определен URL-путь для страницы контактов (например, `'contacts/'`), который связан с контроллером `contacts_view`.

**7. Запуск проекта и доступ к страницам:**

*   Для запуска проекта необходимо перейти в директорию, содержащую файл `manage.py` (обычно это корень проекта `CatalogWeb`), и выполнить команду `python manage.py runserver`.
*   Доступ к домашней странице осуществляется по адресу `http://127.0.0.1:8000/` (или другому адресу, указанному в выводе команды `runserver`).
*   Доступ к странице контактов осуществляется по адресу `http://127.0.0.1:8000/contacts/`.

## Описание выполненной работы: Реализация моделей, работа с медиа, админка и ORM-запросы

Данное описание отражает разработку моделей данных "Продукт" и "Категория" в Django-проекте, настройку работы с медиафайлами, кастомизацию административной панели, а также выполнение ORM-запросов и создание фикстур.

**1. Модели данных:**

*   **Создана модель `Продукт (Product)`:**
    *   Модель `Product` содержит необходимые поля (например, название, описание, цена, изображение, дата создания, дата изменения и т.д.).
*   **Создана модель `Категория (Category)`:**
    *   Модель `Category` содержит необходимые поля (например, название, описание).
*   **Класс `Meta`:**
    *   В моделях `Product` и `Category` описан класс `Meta`, в котором указаны параметры `verbose_name` и `verbose_name_plural` для корректного отображения названий моделей в административной панели.
*   **Метод `__str__`:**
    *   В моделях `Product` и `Category` реализован метод `__str__`, возвращающий строковое представление объекта (например, название).
*   **Связь между моделями:**
    *   Модель `Product` связана с моделью `Category` через поле `ForeignKey`. Это позволяет установить связь "один ко многим" (одна категория может содержать множество продуктов).
*   **Миграции:**
    *   Сделаны и запущены миграции (`python manage.py makemigrations` и `python manage.py migrate`) для создания таблиц в базе данных на основе созданных моделей и полей.

**2. Работа с медиаданными:**

*   **Настройки MEDIA:**
    *   В файле `settings.py` корректно указаны параметры `MEDIA_URL` и `MEDIA_ROOT`.
    *   Настроены URL-адреса для отображения медиаданных в режиме отладки (DEBUG=True) с использованием `urlpatterns += static(settings.MEDIA_URL, document_root=settings.MEDIA_ROOT)`.
*   **Зависимости:**
    *   Установлен и зафиксирован в зависимостях проекта пакет `Pillow` (используется для работы с изображениями) с помощью команды `poetry add --group dev pillow` и записи в `requirements.txt`.

**3. Административная панель:**

*   **Отображение продуктов:**
    *   Для модели `Product` настроено корректное отображение в административной панели Django. Создан класс `ProductAdmin` (унаследованный от `admin.ModelAdmin`) с настройками для отображения полей в списке, полей для редактирования, фильтров и т.д.  Этот класс зарегистрирован в `admin.py` (`admin.site.register(Product, ProductAdmin)`).
*   **Отображение категорий:**
    *   Для модели `Category` настроено корректное отображение в административной панели Django. Создан класс `CategoryAdmin` (унаследованный от `admin.ModelAdmin`) с аналогичными настройками. Этот класс зарегистрирован в `admin.py` (`admin.site.register(Category, CategoryAdmin)`).

**4. Django Shell:**

*   **Установка `ipython`:**
    *   Установлена библиотека `ipython` (для улучшения интерактивного режима Django Shell) и зафиксирована в зависимостях проекта.
*   **Скриншоты:**
    *   Приложены скриншоты терминала, демонстрирующие взаимодействие с Django Shell.
*   **ORM-запросы (create()):**
    *   Созданы категории с помощью ORM-запроса `Category.objects.create()` в Shell.
    *   Созданы продукты с помощью ORM-запроса `Product.objects.create()` в Shell.
*   **ORM-запросы (get all):**
    *   Выполнен запрос на получение всех категорий (`Category.objects.all()`).
    *   Выполнен запрос на получение всех продуктов (`Product.objects.all()`).
*   **ORM-запросы (filter):**
    *   Выполнен запрос на получение всех продуктов определенной категории (`Product.objects.filter(category=...)`).
*   **ORM-запросы (update):**
    *   Выполнен запрос на обновление цены определенного продукта (`Product.objects.filter(id=...).update(price=...)`).
*   **ORM-запросы (delete):**
    *   Выполнен запрос на удаление продукта (`Product.objects.filter(id=...).delete()`).

**5. Фикстуры и кастомные команды:**

*   **Фикстуры:**
    *   Сформированы JSON-фиктуры для моделей `Product` и `Category` (содержащие данные для заполнения базы данных). Фикстуры размещены в директории `fixtures` приложения.
*   **`loaddata`:**
    *   При выполнении команды `python manage.py loaddata <имя_фикстуры>` фикстуры загружаются в базу данных корректно.
    *   При загрузке данных нет проблем с кодировками данных (данные отображаются корректно).
*   **Кастомная команда:**
    *   Реализована кастомная команда для загрузки объектов из фикстур с использованием `BaseCommand`. Команда располагается в модуле `management/commands` внутри приложения.
    *   В кастомной команде происходит предварительное удаление данных перед загрузкой новых данных, чтобы избежать дублирования.
    *   В кастомной команде используются существующие фикстуры.
    *   Связи между продуктами и категориями при загрузке через кастомную команду выстраиваются корректно (поле `ForeignKey` устанавливается верно).
